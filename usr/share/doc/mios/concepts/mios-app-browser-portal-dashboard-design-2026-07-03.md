# MiOS App / Browser / Portal / Dashboard — Design Specification

> AI-hint: Ground-truth design spec for MiOS's user-facing surfaces (Portal,
> Hermes Dashboard, Configurator, Quickshell/Hyprland shell, browser
> integration, Cockpit, mobile Portal). Written after a full audit of both
> repos (`MiOS` + `mios-bootstrap`); corrects prior fabricated claims and
> proposes an SSOT-driven visual unification.
> AI-related: usr/share/mios/mios.toml, usr/share/mios/configurator/mios.html,
> usr/lib/mios/agent-pipe/mios_pipe/routing/portal.py,
> usr/share/mios/quickshell/, usr/share/mios/hyprland/hyprland.conf,
> automation/56-bake-surfer.sh, automation/54-bake-hyprland.sh,
> usr/libexec/mios/mios-sync-theme, usr/share/mios/branding/

## 0. Method

This spec is grounded in a direct read of the `MiOS` repo (system layer) and
`mios-bootstrap` (installer/bootstrap companion), not assumption. Every claim
below cites its source file. A previous draft circulated for this design
contained several fabricated specifics — a "Codex Dark / Pastel Milkshake"
dual-theme, a hand-patched `browser.xhtml` presented as a from-scratch XUL
build, and a ".bp" Blueprint YAML DSL. Section 1 states plainly what's real
and what wasn't, so the rest of the document can be trusted at face value.

**Confirmed fabrications from the prior draft, for the record:**

## 2. Design tokens — the real SSOT palette

The canonical brand palette lives in exactly one place conceptually —
`mios.toml [colors]` — and is mirrored (correctly) in the Configurator's
`:root` block and the Android app's theme resource. This is the **only**
palette that should be called "the MiOS brand":

| Token | Hex | Role |
|---|---|---|
| `bg` | `#282262` | deep indigo — root background |
| `fg` | `#E7DFD3` | foam cream — primary text |
| `accent` / `info` | `#1A407F` | operator blue — panels, links, active state |
| `cursor` / `warning` | `#F35C15` | sunset orange — cursor, warnings, primary CTA |
| `success` | `#3E7765` | wave green — OK/success state |
| `error` | `#DC271B` | coral red — errors |
| `subtle` | `#B7C9D7` | pale blue-grey — borders, dividers |
| `muted` | `#948E8E` | inactive text/borders |
| `silver` | `#E0E0E0` | secondary surface |
| `earth` | `#734F39` | rare, decorative accent |

Source of truth: `usr/share/mios/mios.toml [colors]` (three-layer override:
`~/.config/mios/mios.toml` < `/etc/mios/mios.toml` <
`/usr/share/mios/mios.toml`). `tools/lib/userenv.sh` already exports these as
`MIOS_COLOR_BG`, `MIOS_COLOR_ACCENT`, etc. at every interactive shell start
and — per its own header comment — these are already meant to reach "the
oh-my-posh theme, the configurator HTML's `:root`, and globals.{sh,ps1}."

**The Quickshell panel and the Surfer browser patch do not follow this
convention today** — they hardcode a second, disjoint palette (Catppuccin
Mocha: `#1e1e2e` / `#cdd6f4` / `#a6e3a1`). That is a genuine Law 7
(no-hardcode) gap, not a style choice, because the SSOT values are already
exported and simply unused in those two call sites. §13 fixes both.

Optional/selectable terminal presets (Solarized Dark, Dracula, Nord) already
exist in `mios.html` as user-choosable alternates — those are legitimately
separate from the default brand identity and this spec doesn't touch them.

## 3. Typography

Real and consistent everywhere already: **Geist** (sans) / **Geist Mono**
(monospace), shipped via `usr/share/fontconfig/conf.avail/30-mios-geist.conf`,
with **Symbols Nerd Font Mono** as the icon-glyph fallback. The Configurator's
own font picker names it explicitly: `GeistMono Nerd Font Mono`. There is no
Fira Code or JetBrains Mono anywhere in either repo — don't introduce them.

## 4. Surface: MiOS Portal (the real "Dashboard")

`portal.py` already implements a genuinely well-structured single-page app,
server-rendered (no build step, no framework, offline-first per Law 7):

- A top bar: brand mark, **Install** (PWA), **Chat**, **Settings**, and a
  hamburger **Menu**.
- A dashboard view: a websearch box, an embedded AI chat `<iframe>`, a host
  stats strip, then four sections — **Terminals** (inline 80×20 PTY: bash,
  pwsh, btop), **Swarm Nodes**, **Services** (a card grid with expandable
  per-service detail — URL, container, image, logs), and **MiOS Apps**
  (fuzzy-searchable app launcher).
- A settings view that iframes the Configurator in place.
- Modal sheets, toasts, a login page ("Sign in with your MiOS password"),
  and full PWA plumbing (manifest, icons, service worker) already served from
  `/portal/*`.

This is the right foundation — the aesthetic work is chrome, not
architecture. Concrete, additive proposals (implemented as a drop-in
stylesheet in §13, not a rewrite of `portal.py`'s Python):

- **App-shell window chrome.** Since the Portal is installed as a
  chrome-less PWA (and wrapped as a chrome-less WebView on Android), it owns
  its own title bar. Give it a custom 32px titlebar with **left-aligned,
  macOS-style traffic-light dots** (three small circles: close/minimize/
  maximize, using `error` / `warning` / `success` tokens for their fill) —
  this reads immediately as "this is an app, not a browser tab," which is
  the actual intent of installing it as a PWA in the first place.
- **Vertical tab rail**, not a horizontal top-bar-only nav: turn the
  existing Terminals / Swarm / Services / Apps sections into a left-anchored
  icon rail (48–56px wide, expandable to show labels on hover/focus) so the
  dashboard reads as a single persistent workspace rather than a scrolling
  page. This matches the operator's existing mental model from the terminal
  side (`mios.toml [theme]` already ships a frameless, scrollbar-hidden,
  acrylic-backdrop terminal profile — the GUI should feel like the same
  object, not a different one).
- **Frosted/translucent panels** with a thin (1px, `subtle` token) border and
  8–12px rounded corners on every card — this mirrors, not duplicates, what
  Hyprland's compositor already does for windows (`decoration.rounding = 10`,
  `blur.enabled = true` — confirmed in `hyprland.conf`); the web content
  should look native to the compositor it's floating inside of.
- **Hi-DPI**: the Portal is plain server-rendered HTML/CSS already, so this
  is a matter of using `rem`/viewport units and `image-set()`/SVG icons
  rather than fixed px + raster PNGs where practical; the existing
  `icon-192.png`/`icon-512.png` PWA icon pair already covers the manifest
  requirement.

## 5. Surface: Hermes Dashboard

A second, legitimately separate dashboard (kanban + `/chat` bash terminal +
OpenUI generative-UI panel), served by `hermes-dashboard.service`, which
degrades to a plain explainer page (`web_dist_stub/index.html`) when the
React SPA isn't built from source. It should **not** be visually merged into
the Portal — they're different services with different failure modes (the
Portal is a first-class always-on surface; the Hermes Dashboard is
best-effort and explicitly allowed to degrade to a stub). What they should
share is the token file from §2/§13, so that when the SPA *is* built, it
picks up the same brand palette instead of Nous Research's upstream default
theme.

## 6. Surface: Configurator (`mios.html`)

Already the palette's source of truth in HTML form and already offline-first
(local `@font-face`, no CDN). No correction needed here beyond adopting the
shared `mios-app-shell.css` chrome classes from §13 for its own window
frame, so that opening it standalone (rather than iframed inside the Portal's
settings view) still looks like the same app family.

## 7. Surface: Desktop shell — Hyprland + Quickshell

**What's already right:** `hyprland.conf` already enables rounded corners
(`rounding = 10`), real compositor-level blur (`blur.enabled = true`, 3
passes, xray mode — genuine frosted glass, not a CSS approximation), and
drop shadows. The frosted/rounded aesthetic the operator wants is already
turned on at the compositor level; it just isn't color-matched.

**What's wrong:** `col.active_border = rgba(33ccffee) rgba(00ff99ee) 45deg`
is a neon cyan→green gradient with no relationship to the brand palette, and
the Quickshell panel hardcodes Catppuccin Mocha. Both are fixed in §13 by
sourcing `MIOS_COLOR_*` (already exported, already unused at these two call
sites) instead of introducing new literals.

**Proposed redesign of the Quickshell panel** (implemented in §13):
a frameless top strip (unchanged 36px height) plus a new left-anchored
vertical launcher rail — workspace indicator, agent-status pill, a
**Cockpit quick-launch tile** (opens `http://localhost:9090` in the default
browser — Cockpit is real and already wired, see §9, so this is one line),
and the clock — all reading colors from a small `Theme.qml` bridge component
instead of inline hex, so a future palette change in `mios.toml` propagates
here without touching QML.

## 8. Surface: Browser integration — reconciling three paths

The repo currently ships three overlapping, non-communicating ideas of "the
MiOS browser":

1. **`mios-webshell`** (`automation/56-bake-surfer.sh`) — a real, working
   build step: clones `zen-browser/surfer`, pulls upstream Firefox via
   `npx surfer download`, patches `browser.xhtml` with a 3-pane layout
   (`mios-custom-sidebar` / `appcontent` / `mios-ai-sidebar`), compiles, and
   symlinks the result to `/usr/bin/mios-webshell`. Hyprland already has a
   window rule floating and sizing it (`hyprland.conf` lines ~74–76) and a
   `$mainMod, E` keybind to launch it.
2. **Zen Browser via winget** (`mios.toml [browser_ai]`, implemented in
   `mios-bootstrap/src/install-host-tools.ps1::Configure-MiosBrowserAI`) — a
   *prebuilt* Zen Browser (Twilight channel) on the **Windows host**, wired
   to the local AI endpoint through Firefox's native `browser.ml.chat.*`
   Smart Window prefs. `mios-bootstrap/src/mios-app/RESEARCH.md` explicitly
   evaluated and *rejected* a from-source Surfer build on the Windows side
   (30–40 GB build, disproportionate) in favor of exactly this path.
3. **Flatpak GNOME Epiphany + Chrome Dev** (`mios.toml [[desktop.apps]]`,
   `role = "browser"`) — the Linux-VM-side browsers; Chrome Dev doubles as
   the CDP target for Hermes's own `browser_*` tools.

These are not contradictory so much as **scoped to different hosts** and
should be documented as such rather than merged: (1) is the bare-metal/VM
in-session browser shell; (2) is the Windows-host companion browser for
operators running the WSL2/dev-VM topology; (3) is the flatpak fallback
inside the Linux desktop session for everyday browsing + agent CDP control.
The one concrete defect worth fixing is (1)'s color literals — done in §13
so `mios-webshell`'s sidebar finally matches the rest of the OS.

## 9. Surface: Cockpit integration

Real, not aspirational. `cockpit.socket` is enabled directly; a rootless
Quadlet (`mios-cockpit-link.container`, `socat`) relays a configurable port
to `host.containers.internal:9090` for the WSL2 dev-VM topology, and
`usr/lib/sysusers.d/50-mios-cockpit.conf` pins static UIDs for
`cockpit-ws`/`wsinstance` workers because WSL2 disables `DynamicUser=`. The
Portal's **Services** grid already discovers and cards up running
services generically (`_discover_portal_services()` in `portal.py`) — Cockpit
should simply appear there like any other service once its Quadlet is
running, rather than needing bespoke UI. The Quickshell quick-launch tile
in §7/§13 covers the "one click from the desktop shell" case.

## 10. YAML / Quadlet conventions

No new DSL is being introduced (there is no ".bp" format in this repo and
none is proposed). The existing Quadlet convention already covers
"declarative app" needs — a `.container` unit with `User=`/`Group=`/
`Delegate=yes` per Law 6, `PublishPort=` for the port, and a bind-mount for
any read-only data it needs. `mios-cockpit-link.container` is the reference
example to imitate for any future "browser-facing service front door" quadlet.

## 11. SSOT cohesion for offline/localhost config + deploy

The three-layer `mios.toml` override (`~/.config` < `/etc` < `/usr/share`),
the `mios-sync-env` → `/etc/mios/install.env` bridge, and `MIOS_AI_ENDPOINT`
(Law 5) are the existing, working pattern for "everything reads from one
file, everything works offline, nothing points at a vendor cloud URL." This
spec's only addition (§13) is a sibling bridge, `mios-sync-theme` →
`/etc/mios/theme/{theme.json,mios-theme.css}`, following the exact same
shape so CSS/QML consumers get the same one-file-changes-everything property
that shell/systemd consumers already have via `install.env`.

## 12. What shipped in this pass (see repo diff)

- `usr/libexec/mios/mios-sync-theme` — new generator, reads `mios.toml`
  `[colors]` + `[theme.font]`, writes the `/etc/mios/theme/` bridge.
- `usr/lib/tmpfiles.d/mios-theme.conf` — declares `/etc/mios/theme` (Law 2).
- `usr/lib/systemd/system/mios-sync-theme.service` — oneshot runner.
- `usr/share/mios/quickshell/Theme.qml` — new bridge component (FileView
  over the generated `theme.json`, degrade-open fallback = current vendor
  palette).
- `usr/share/mios/quickshell/Config.qml`, `PanelWindow.qml` — redesigned:
  vertical launcher rail, Cockpit tile, SSOT colors via `Theme.qml`.
- `usr/share/mios/branding/mios-app-shell.css` — reusable chrome classes
  (frosted panel, left-aligned traffic-light window controls, vertical tab
  rail, thin borders, rounded corners) for the Portal/Configurator/Hermes
  Dashboard to opt into.
- `automation/56-bake-surfer.sh` — 3 hardcoded Catppuccin literals replaced
  with `MIOS_COLOR_*` (sourced via `lib/common.sh`, already the repo-wide
  convention every other automation script uses).
- `automation/54-bake-hyprland.sh` — border colors now sourced from
  `MIOS_COLOR_ACCENT`/`MIOS_COLOR_INFO`/`MIOS_COLOR_MUTED` via a placeholder
  + `sed` substitution (the heredoc stays single-quoted on purpose — it also
  contains literal `$mainMod` Hyprland variable references that must **not**
  be shell-expanded); added a Cockpit quick-launch keybind.

## 13. Remaining gaps / follow-ups (not done in this pass)

- `mios-sync-theme` is not yet wired into `automation/build.sh`'s numbered
  pipeline or a `mios update` path — it exists and is safe to run by hand,
  but for the bridge file to be present on first boot it should be invoked
  once during Phase-2 (a natural home is right after `30-locale-theme.sh`)
  and again by a path-unit watching `mios.toml`, mirroring `mios-sync-env`.
- The Hermes Dashboard's React SPA (when built) should import
  `mios-app-shell.css` — that's a change in the *upstream* `hermes-agent`
  `web/` source tree, out of scope for this repo.
- `mios-webshell`'s AI sidebar currently points at `http://localhost:3030`
  (OWUI) inside the patch; confirm this is still the intended target versus
  `agent-pipe:8640/v1` before the next Surfer rebuild — the two browser
  paths in §8 currently disagree on which port is "the" AI endpoint.
