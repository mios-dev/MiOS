> Status: architecture doc, final for MiOS DOCS, mid-2026. Merges the mid-2026 draft with the critic pass. All FOSS claims are deep-read-confirmed; every load-bearing claim carries its source URL inline. Both planes are `mios.toml`-SSOT-projected bootc images built off one toolchain. ## 0. Critic-driven changes (what changed from the draft)

This revision fixes four load-bearing errors and closes seven gaps the critic surfaced:

1. **GPU fractioning is now honestly scoped as impossible on the driver-free host.** The draft's rescue path — "SR-IOV VFs bindable to `vfio-pci` *without* a host GPU driver" — is factually wrong: AMD GIM, NVIDIA vGPU-host, and Intel Xe SR-IOV all load a **host-side PF driver** to spawn VFs ([AMD GIM](https://github.com/amd/MxGPU-Virtualization), [Intel xe SR-IOV](https://docs.kernel.org/gpu/rfc/xe_sriov.html)). §3a #7 is rewritten: whole-GPU-to-one-guest is the *only* path that preserves the driver-free-host invariant; multi-guest fractioning is retained-but-gated as a **conscious invariant-breaking opt-in**, with the honest conclusion stated.
2. **The firewalld-vs-hand-authored-nftables contradiction is resolved** in favor of a hand-authored multi-table `inet` design with **no firewalld**; libvirt/netavark/tailscale each own their *own* nftables tables that coexist by hook priority (§2a, §3b-cross). The draft asserted both firewalld-mandatory and a bespoke master table — mutually exclusive.
3. **swtpm-emulated vTPM is added as the guest-FDE default** (the upstream-standard VM-TPM answer), enabling in-guest `clevis-tpm2` measured into guest PCRs; Tang-NBDE is demoted to a *fleet-only* option with its node-location problem stated plainly (§2b, §3a #2).
4. **Property #1 ("tiny host") is quantified and reconciled, not merely regretted** (§1.1), and three previously-missing sections are added: **storage-silicon ownership** (§2c), **headless recovery** (§2d), and the **router core budget** (§2e). The **APU-only MiOS-Xbox** worst case is addressed honestly (§2d, R2/R10). **Gluster gets its own sunset writeup** (§3a #5).

## 1. Executive summary

**MiOS-Mini** is a deliberately small Fedora **bootc** image whose only job is to be the *metal-owning firmware* of a box: a headless, immutable **hypervisor + hardware packet-router + Wi-Fi access point**. It owns the NICs, the radios, the firmware, the TPM, the boot/root storage, and the boot chain — and as little else as a hypervisor floor allows. Everything that looks like an "operating system" to a user or workload lives in the **MiOS guest**: the full MiOS AI-OS OCI image, booted by MiOS-Mini as a **super-privileged VM** handed **all discrete GPUs and ~75-90% of CPU/RAM**, but given **no NIC of its own**. The guest reaches the world exclusively by being routed through the host.

The split dissolves a real conflict. MiOS wants a rich GPU-accelerated desktop shell (Hyprland/Quickshell liquid-glass), which needs a GPU bound to a display driver; MiOS also wants *every* GPU available to ROCm/CUDA compute. On one box those demands fight over the card. The split-plane makes the fight disappear **when at least one discrete GPU exists**: the router host is headless by design, so it binds every discrete GPU to `vfio-pci` and consumes none ([kernel VFIO docs](https://docs.kernel.org/driver-api/vfio.html)). With zero host GPU consumers, 100% of display *and* compute silicon belongs to the guest. This premise has a hard boundary — **APU-only, single-iGPU boxes (the MiOS-Xbox worst case)** — treated explicitly in §2d and R10 rather than assumed away.

Three properties make the design coherent:

- **One toolchain, two planes.** Host and guest are both bootc images produced by `bootc` ([bootc-dev/bootc](https://github.com/bootc-dev/bootc), Apache-2.0 OR MIT, v1.16.4, 2026-07-15) and baked by `bootc-image-builder` ([osbuild/bootc-image-builder](https://github.com/osbuild/bootc-image-builder), Apache-2.0). Host installs to bare metal; guest is baked to a qcow2 the host boots. Both projected from `mios.toml`. Cross-reference: [MiOS Roadmap](file:///C:/MiOS/ROADMAP.md) and [MiOS Deploy Model](file:///C:/MiOS/usr/share/doc/mios/concepts/deploy-model.md).
- **The host owns the bind; the guest owns the silicon.** `vfio-pci` at early boot is the clean line: the host loads no GPU driver, and libvirt `<hostdev>` hands whole IOMMU groups to the guest.
- **The host owns the network; the guest owns nothing on the wire.** MiOS-Mini runs the **headscale** control plane ([juanfont/headscale](https://github.com/juanfont/headscale), BSD-3-Clause, v0.29.2, 2026-07-01), is the sole subnet-router/exit-node, and forwards 100% of guest traffic on the kernel `nftables`/flowtable fast path.

```
                         ┌─────────────────────────────────────────────┐
   WAN / uplink ─── NIC ─┤  MiOS-MINI  (small bootc host, headless)     │
   Wi-Fi 7 radio ── PHY ─┤  hostapd · nft inet(+flowtable) · headscale  │
                         │  tailscaled (subnet-router/exit) · dnsmasq   │
                         │  libvirt/KVM · driverctl→vfio-pci ALL dGPUs  │
                         │  owns: NICs·radios·TPM·boot/root NVMe·/var    │
                         │      │ virtio-net (only path to the world)   │
                         │      ▼   <hostdev> ALL dGPUs + 75-90% CPU/RAM│
                         │  ┌──────────────────────────────────────┐   │
                         │  │  MiOS GUEST (full AI-OS bootc image)  │   │
                         │  │  ROCm/CUDA · k3s · ceph · desktop     │   │
                         │  │  swtpm vTPM · clevis-tpm2 FDE         │   │
                         │  │  NO host NIC — egress via host mesh   │   │
                         │  └──────────────────────────────────────┘   │
                         └─────────────────────────────────────────────┘
```

### 1.1 On property #1 ("tiny host") — quantified, not hand-waved

The north-star property is: *"MiOS-Mini is TINY: a handful of scripts or fewer, NO big pulls/deps atop the minimal base."* The critic is correct that a hypervisor-router cannot be literally "a handful of scripts": a KVM host has an **irreducible floor** of `libvirtd` + `qemu-kvm` + `edk2/OVMF`, plus the routing set (`nftables`, `hostapd`, `dnsmasq`, `headscale`, `tailscaled`). We reconcile the property honestly rather than regret it:

- **"Handful of scripts" is reinterpreted as the *config/automation surface*, not the dependency count.** MiOS-Mini adds **zero hand-maintained config files**: every host surface (UKI cmdline, `/etc/driverctl.d`, the nft ruleset, `hostapd.conf`, `dnsmasq.conf`, headscale policy, libvirt domain XML) is *projected from `mios.toml` and drift-gated* (§4). The operator authors `mios.toml`, not a pile of scripts. That is the property that actually holds.
- **The dependency floor is enumerated and each entry justified** (§2a); nothing is on the host that is not required by *routing* or *hypervision*. Everything else — ROCm/CUDA userspace, k3s, ceph, desktop, `nvidia-ctk`, `bootc-image-builder`, `virt-v2v` — is build-time or guest-plane.
- **Rough size budget (indicative, `fedora-bootc-minimal` base ≈ 400-600 MB):** host floor adds ≈ **900 MB-1.4 GB** (libvirt+qemu+OVMF ≈ 500-800 MB dominant; headscale+tailscale ≈ 60 MB; hostapd/dnsmasq/nftables/driverctl ≈ 15 MB; vendor-reset akmod ≈ negligible). The **full MiOS guest image is ~8-15×** larger (ROCm alone is multi-GB). "Tiny" is therefore *relative and real* — the host is an order of magnitude smaller than the guest — but the literal "handful of deps" reading is **not met and is not achievable for a KVM host**; this is documented as R1, now with a number attached.

### 2c. Storage-silicon ownership (NEW — plane boundary the draft left undefined)

The plane split named NICs/radios/TPM/firmware as host-owned but never assigned **NVMe/SATA/HBA controllers**. This is resolved by box-tier:

- **Host always owns the boot/root device** (the NVMe/SATA holding the bootc root + `/var`). `/var` holds the guest qcow2, headscale DB, dnsmasq leases, swtpm NVRAM, vendor-reset state — all non-negotiably host-local.
- **Multi-drive boxes (workstation/server tier): pass a *dedicated data NVMe/HBA* through to the guest** via `<hostdev>` (its own IOMMU group), giving guest ceph/model-cache **fast raw block** with no double indirection. Host boot disk and guest data disk are physically separate devices. This is the preferred topology and the SSOT default when `[storage.guest_passthrough]` names ≥1 non-boot controller.
- **Single-drive boxes (MiOS-Xbox tier): guest storage is `virtio-blk`/`virtio-scsi` on a host-backed volume** (raw image or LVM thin LV on host `/var`). This accepts one layer of indirection (guest ceph OSD on a virtio device backed by a host file) in exchange for not stranding host `/var`. ceph on a single virtio-backed OSD is a degenerate 1-node cluster (§3a #5) — acceptable for a single box, distributed value appears only in a fleet.
- **Never pass the boot NVMe to the guest** — it strands host `/var`. The drift-gate validates that `[storage.guest_passthrough]` never names the boot controller's BDF.

### 2d. Headless recovery & the APU-only worst case (NEW)

Binding **all** discrete GPUs via `vfio-pci.ids` at early boot means **no efifb/simpledrm console** — the host has no video. Combined with FDE lockout on firmware/UKI churn (R3) and no BMC/serial on consumer hardware, a failure can yield a box with no video *and* no network = a physical-reseat brick. The draft treated these as separate risks; their **intersection removes the last recovery channel**. Mitigations, layered:

1. **Always an escrowed off-box recovery passphrase** on the host LUKS (step 3) — never rely on TPM-only unlock.
2. **A recovery UKI boot entry that does NOT bind vfio-pci** (SSOT-projected second entry): boots the host with `amdgpu`/`i915` claiming a GPU for a real console, no passthrough, network up. Selectable from the boot menu on physical presence. This is the headless router's "safe mode."
3. **Serial console as first-class** where the board exposes COM/USB-serial (`console=ttyS0` in the recovery cmdline) — cheap, board-dependent, the router's natural out-of-band channel.
4. **On multi-GPU boxes, optionally reserve the weakest GPU (or the iGPU) for host efifb** rather than binding literally every card — a small compute sacrifice for a permanent console. SSOT `[gpu.host_console_reserve]`.

**APU-only / single-iGPU MiOS-Xbox — the founding premise's hard boundary (honest).** The "split dissolves the single-GPU conflict" argument *assumes ≥1 discrete GPU to hand over while the host stays headless.* On an APU-only box:
- The iGPU typically sits in the **root-complex IOMMU group with host-critical devices** (R2), so clean passthrough is often **impossible without ACS-override** (a security downgrade on a router).
- Passing it (where possible) leaves the host with **no display at all** and no fallback console.
- **Conclusion (not assumed away):** on APU-only hardware the split-plane **degrades to one of two documented fallbacks**, chosen in `mios.toml`: **(a) single-plane MiOS** — run the full MiOS bootc image *directly on metal* (no MiOS-Mini split), accepting the single-GPU-vs-desktop conflict the split was meant to dissolve; or **(b) headless-compute split** — keep the split, leave the iGPU on the host for a console, and give the guest **no GPU** (CPU-only or remote-GPU compute). The MiOS-Xbox derivative is the case this architecture serves *worst*, and it is labeled as such (R10) rather than presented as the motivating win.

### 2e. Router core budget (NEW — the 75-90%-to-guest claim, sized)

The draft asserted 75-90% of CPU/RAM to the guest without showing the router survives at the top of that envelope. Concretely:

- **HW flowtable offload requires a NIC implementing `TC_SETUP_FT`** ([nf_flowtable](https://docs.kernel.org/networking/nf_flowtable.html)) — most consumer/homelab NICs do **not**, so the **software fast-path runs on the host housekeeping slice** (the 10-25% not given to the guest).
- **Budgeting rule (SSOT-enforced):** reserve the housekeeping/`isolcpus`-excluded slice for the *measured* worst-case forwarding load, sized **before** allocating the guest's share — line-rate first, then the guest gets the remainder. Indicative floors: **≥2 physical cores + 2-4 GB** reserved for the router path on a Wi-Fi-7 / dual-2.5GbE homelab box; **≥4 cores** if aggregating 10GbE without HW offload (SW flowtable sustains multi-Gbps per core but WPA3/hostapd + conntrack + nftables + tailscaled WireGuard all draw from the same slice). On an 8-core box that is the 75% figure, not 90%; **90% is only safe with a `TC_SETUP_FT`-capable NIC** doing HW offload so the CPU path is near-idle. The SSOT projection **computes the guest share from the reserved router floor**, and the drift-gate fails if the guest allocation would leave the router under its declared floor.

#### Writeup — capability #7 (mdevctl / GPU fractioning): why it is technically impossible on the driver-free host — CORRECTED

**The MiOS component is NOT dropped:** `mdevctl` stays packaged and `[gpu.mdev]` remains a valid SSOT surface. What is impossible is the *north-star combination*: **any GPU fractioning on a host that loads no GPU driver.** The draft's rescue path (SR-IOV VFs "without a host GPU driver") was **factually wrong** and is retracted here.

- **Why mdev can't work as-is.** mdev requires the *parent* GPU to expose mdev types from a **host driver**. NVIDIA mdev/vGPU types come only from the **proprietary vGPU Manager host driver** (not in [open-gpu-kernel-modules](https://github.com/NVIDIA/open-gpu-kernel-modules)); Intel's legacy **GVT-g** mdev path is deprecated/removed on newer kernels. On the open/consumer stack there are frequently **no mdev types to create at all**.
- **Why SR-IOV does NOT rescue it (the corrected core fact).** Creating SR-IOV virtual functions on a GPU requires a **host-resident PF (physical-function) driver**: **AMD needs `amdgpu` + the GIM host module** ([amd/MxGPU-Virtualization/GIM](https://github.com/amd/MxGPU-Virtualization)); **NVIDIA needs the vGPU host driver**; **Intel needs the `xe` SR-IOV PF driver** ([xe SR-IOV](https://docs.kernel.org/gpu/rfc/xe_sriov.html)). In every case a **GPU driver is loaded on MiOS-Mini** to spawn the VFs — the exact "host loads no GPU driver, every GPU on `vfio-pci`" invariant the whole split rests on. **SR-IOV therefore does not preserve the driver-free host; it violates it identically to mdev.**
- **Honest conclusion.** On the north-star driver-free-host stack, the **only** supported GPU topology is **whole-GPU-to-one-guest via `vfio-pci`**. Multi-guest / management-sliver **fractioning is unsupported as long as the driver-free-host invariant holds — period.** It is retained only as a **conscious, operator-enabled opt-in that knowingly breaks the invariant**: an operator may install a PF/vGPU host driver (GIM / NVIDIA vGPU / xe-SR-IOV) on MiOS-Mini to expose VFs (then bindable to `vfio-pci` for guests), accepting a **documented, non-silent erosion** of the headless-router-owns-no-GPU-driver guarantee (and, for NVIDIA, a proprietary driver on the router). `mdevctl` remains the persistence layer *only* where such a parent genuinely exposes mdev/SR-IOV types. **Net:** default = whole-GPU passthrough; fractioning = invariant-breaking opt-in, flagged loudly — not a phantom "SR-IOV keeps the invariant" retention.

#### Writeup — capability #5b (Gluster): keep-with-sunset, because upstream is retired

**Gluster is NOT silently dropped.** But it warrants its own decision because it is **upstream EOL/retired** (Red Hat Gluster Storage retired; upstream community activity wound down), which the draft buried in a one-line demotion.

- **Why not keep it as a peer of ceph.** Filing an upstream-retired dependency as plain "adapt-to-align" alongside actively-maintained standards (chrony/nut) is a mismap — it will receive no security fixes going forward and cannot be a *go-forward* choice.
- **Why not drop it outright.** Existing MiOS deployments may hold live Gluster volumes; a silent kill would strand data.
- **Decision.** **`keep-MiOS-way (sunset)`:** Gluster is retained **read-mostly, for existing volumes only**, packaged in the guest, with **Ceph as the sole go-forward path** for all new storage. `mios.toml` marks `[storage.gluster]` as `deprecated=true`; the drift-gate emits a **sunset warning** on any new Gluster volume definition and the docs steer migration to Ceph. This is an explicit keep-or-kill decision with a stated end-state, not a bucket collision.

KubeVirt pulls a full k3s+KubeVirt control plane onto a host mandated tiny — **it belongs in the guest orchestration plane**, not on MiOS-Mini. Plugin is NVIDIA-only; AMD/Intel need GPU-Operator's generic VFIO manager or plain libvirt. |

**Cross-cutting sub-decisions — now decided, not deferred:**

- **Host firewall ownership — RESOLVED: hand-authored multi-table nftables, NO firewalld.** The draft's firewalld-mandatory claim contradicted its own hand-authored master-table design and cannot express `flow add @ft`/`flags offload` fast-path rules. **Resolution:** MiOS-Mini authors **one master `inet mios-router` table** (WAN masquerade, default-drop forward, per-bridge isolation, flowtable) via `nft -f`, projected from `mios.toml`. The **other nft writers each own their own separate table and coexist by hook priority** — nftables' native multi-table model, no single arbiter required: libvirt uses its **native nftables backend** (`firewall_backend=nftables`, its own `ip libvirt`/`inet libvirt` table); netavark writes `inet netavark`; **tailscaled writes its own `ts-*` chains/table** ([tailscale nftables mode](https://github.com/tailscale/tailscale)). The four writers do **not** share a table, so they do not fight for ownership; the drift-gate validates that no two tables install conflicting masquerade/forward verdicts on the guest bridge, and that conntrack sees both directions across the bridged path. [OpenWrt 24.10/fw4](https://github.com/openwrt/openwrt) remains a **design reference** for the single-table pattern, not a dependency. This is the critic's #2 hard point resolved: the master table is hand-owned; firewalld is removed rather than declared mandatory; tailscale's self-management is the fourth writer, reconciled by table-separation.
- **MagicDNS vs dnsmasq — RESOLVED (a rule, not "decide per segment"): dnsmasq is authoritative for the guest bridge; MagicDNS is split-forwarded.** `dnsmasq` owns DHCP + DNS on the guest-facing bridge. Tailnet (`*.ts.net` MagicDNS) names are handled by a **split-DNS forward**: dnsmasq forwards the tailnet domain to the local tailscaled/headscale resolver and answers everything else itself. One owner per query class, deterministic, drift-gateable — the collision the draft left open is now a fixed rule.
- **Tailscale routing mode** — [netstack/gVisor userspace](https://github.com/tailscale/tailscale/tree/main/wgengine/netstack) vs kernel. Userspace avoids `NET_ADMIN`/`ip_forward` (smaller privilege) but is TCP/UDP-only and lower-throughput; **default kernel** for full-bandwidth GPU-guest egress and all IP protocols. SSOT-projected setting.
- **Guest identity granularity (mini-gap, acknowledged).** With tailscaled on the *host* as subnet-router, the guest is a **routed subnet, not a tailnet node** — Grants apply to the *route*, not to a per-guest identity. Step 9's verification is scoped accordingly. Per-guest node identity would require tailscaled *inside* the guest, which reintroduces guest network autonomy and is therefore **not** the default; documented as a deliberate granularity trade, not overstated.

## 4. Implementation plan

Ordered, buildable steps. Each artifact is projected from `mios.toml` and drift-gated.

1. **Fork the host base image.** Derived bootc image from `fedora-bootc-minimal` (D1). Layer: `libvirt`(≥12.2.0)+`qemu-kvm`+`edk2/OVMF`+`swtpm`, `hostapd` 2.11 (image-baked for Wi-Fi 7), `driverctl`, `dnsmasq`, `nftables`, `chrony`, `nut` (master), `headscale`+`tailscale`, and (if AMD passthrough) `vendor-reset` akmod. **No firewalld.** Keep the layer set as small as the split allows (§1.1 budget). Build with [bootc-image-builder](https://github.com/osbuild/bootc-image-builder).
2. **Bake the signed boot chain (host SB domain).** Generate the UKI with [`systemd-ukify`](https://github.com/systemd/systemd) carrying SSOT kargs: `intel_iommu=on`/`amd_iommu=on iommu=pt`, `vfio-pci.ids=<all dGPU BDFs>` (+ initramfs ids for NVIDIA modeset), `isolcpus`/`nohz_full`/`default_hugepagesz`/hugepage count **sized from the router floor (§2e)**. **Also bake a second, non-vfio recovery UKI entry** (§2d). Ship host Secure Boot keys via [`sbctl`](https://github.com/Foxboron/sbctl) where firmware allows owner PK, [`bootctl`](https://github.com/systemd/systemd/tree/main/src/boot) auto-enroll on setup-mode firmware, [shim 16.1](https://github.com/rhboot/shim)+MOK as locked-firmware fallback. **This is the HOST SB domain — distinct from the guest's OVMF/MOK (step 6).** Drift-gate the on-ESP `.cmdline` against `mios.toml`.
3. **Install to metal + enroll host FDE.** `bootc install to-disk`. Re-enroll LUKS with [`systemd-cryptenroll --tpm2-public-key-pcrs=11`](https://github.com/systemd/systemd) (signed-policy on the UKI-measured PCR) **plus an escrowed off-box recovery passphrase** — do NOT rely on bootc's default `--block-setup tpm2-luks` (no recovery slot → headless lockout, R3/§2d). Layer [`systemd-pcrlock make-policy`](https://github.com/systemd/systemd); sequence `fwupd` capsule-apply → `pcrlock make-policy` as an SSOT update hook (until [fwupd #6318](https://github.com/fwupd/fwupd) lands). Host uses the **physical** TPM.
4. **Verify the bind.** Confirm every discrete GPU (+audio/bridge functions) is on `vfio-pci` and no `amdgpu`/`nvidia`/`i915` claimed a passthrough card (respecting `[gpu.host_console_reserve]` if set, §2d). Project `[gpu.vfio_overrides]` → `/etc/driverctl.d` + post-boot drift check (slot swap must fail loud).
5. **Assign storage silicon.** Per §2c: validate `[storage.guest_passthrough]` never names the boot controller; on multi-drive boxes `<hostdev>` the data NVMe/HBA to the guest; on single-drive boxes provision a virtio-backed guest volume on host `/var`.
6. **Bake the guest image (guest SB domain).** Build the full MiOS bootc image: `nvidia-open` (akmods + GSP firmware **signed into the guest's OWN OVMF nvram/MOK**), ROCm/Intel-compute, k3s, ceph (+ sunset gluster read-path), desktop shell, `nvidia-container-toolkit`, `clevis`(tpm2). Emit qcow2 via bootc-image-builder. Pin driver versions in the guest SBOM (SBOM-not-hardcode). Cross-reference: [MiOS Architecture](file:///C:/MiOS/usr/share/doc/mios/concepts/architecture.md) and [Unified AI Pipeline](file:///C:/MiOS/usr/share/doc/mios/concepts/unified-ai-pipeline-2026-06-16.md).
7. **Define the guest domain.** Project libvirt domain XML from `mios.toml`: `<hostdev>` for all dGPU IOMMU groups (+ data storage from step 5), `host-passthrough` CPU, `<vcpupin>`/`<emulatorpin>`, `<hugepages>`+`<locked/>`, `numatune strict` on the GPU's node, **`<tpm backend='emulator' version='2.0'>` (swtpm vTPM)**, and a single `virtio-net` to the host bridge (the guest's only path — **no second NIC**). Guest allocation computed from the router floor (§2e). Ship as a Quadlet/libvirt unit. Optionally select Cloud Hypervisor (D2).
8. **Enroll guest FDE.** In-guest `clevis-tpm2` (or `systemd-cryptenroll --tpm2`) against the **emulated vTPM** unlocks guest model/data volumes offline (§2b) — plus a guest-side escrowed recovery key. Tang-NBDE only if a fleet peer Tang exists (do not co-locate on this host).
9. **Wire guest-internal GPU consumption.** In the guest, `mios gpu cdi-regen` (`nvidia-ctk cdi generate`) at first-boot/driver-change → `/etc/cdi/nvidia.yaml`; Quadlet/k3s reference `nvidia.com/gpu=all`. Drift-gate the CDI output.
10. **Stand up the router plane.** Load the master `inet mios-router` table (masquerade to WAN, default-drop forward, per-bridge isolation, flowtable). Bring up `hostapd` (country/EHT/WPA3-SAE from SSOT) and `dnsmasq` on the guest bridge only, with the **split-DNS MagicDNS forward rule** (§3b). Start `headscale` (Quadlet, **DERP disabled single-box**) with a **fail-closed restrictive Grants policy** (never allow-all) and `tailscaled` subnet-router/exit-node with `autoApprovers` on `tag:router`. All state on `/var`. Verify libvirt/netavark/tailscale each own their own nft table without conflicting verdicts.
11. **Close the network loop.** Confirm the guest has **no route to the world except through the host**: default route via host bridge, egress via headscale exit node, DNS via host dnsmasq. Verify the Grants policy gates the guest's **route** to only permitted destinations (route-level, per §3b identity note).
12. **Install the DR state-guard (OS + GPU-wedge aware).** Since `bootc rollback` reverts the OS but not `/var`, bolt on a restic/snapshot + schema-version check with [snapper-style](https://github.com/openSUSE/snapper) subvolume-exclusion: decide which `/var` (small router config) rolls WITH the OS vs which (guest qcow2, model cache, headscale DB, swtpm NVRAM) is pinned/guarded. **Wire the greenboot↔vendor-reset interaction (R5):** a guest health-check rollback must run the libvirt teardown hookscript (vendor-reset) *before* re-launch, and if the card fails to reset, fail the guest start loudly rather than wedge silently. Pair every host OS rollback with a matching `/var` restore point (R4).
13. **Wire the drift-gates.** Extend MiOS drift-checks for: UKI `.cmdline` (both entries) vs SSOT, `/etc/driverctl.d` bind list, the master nft table + no cross-table verdict conflict, hostapd/dnsmasq/headscale-policy projections, split-DNS rule, `/etc/cdi` guest spec, storage-passthrough-never-boot-disk, guest-share-≥-router-floor, gluster-sunset warning, and the **fail-closed** headscale policy. No hand-maintained config anywhere.

## 5. Open risks

- **R1 — Host-size vs virt/router weight (the central tension, now quantified).** The host carries the irreducible libvirtd+qemu+OVMF+swtpm floor plus the routing set — ≈ 0.9-1.4 GB atop the base (§1.1), an order of magnitude smaller than the guest but **not** "a handful of deps." Property #1 holds only as a *config-surface* claim (zero hand-maintained config), not a dependency-count claim. Mitigation: relentless layer minimization, Cloud Hypervisor as a lighter VMM (D2), strict build-time-only for bootc-image-builder/virt-v2v.
- **R2 — IOMMU grouping is hardware-dictated.** On cheap consumer boards a GPU can share a group with a host-needed device → all-or-nothing passthrough or the **ACS-override patch (a real security downgrade)** on an immutable router. No software fix; a hardware-selection constraint. Acute on APU-only boxes (R10).
- **R3 — Headless lockout from FDE + firmware churn, intersecting with all-GPU bind.** A capsule (PCR0/PCR4) or unsigned/local UKI can drop the router to a passphrase prompt with **no operator present** — and with all GPUs on vfio there is **no video console either** (§2d). `systemd-pcrlock` is experimental (v255→v258 differ); fwupd↔pcrlock ([#6318](https://github.com/fwupd/fwupd)) unmerged. Mitigation: signed-policy PCR11 + pcrlock + **escrowed off-box recovery key** + **non-vfio recovery UKI entry** + serial console (§2d); gate capsule apply behind confirmed PCR re-prediction.
- **R4 — Rollback data-plane skew.** `bootc rollback` moves the OS back but `/var` (qcow2, model cache, headscale DB, swtpm NVRAM) stays forward; bootc ships no `/var` guard. Mitigation: the external state-guard (step 12) is mandatory.
- **R5 — AMD reset & GPU re-provisioning on guest lifecycle.** [vendor-reset](https://github.com/gnif/vendor-reset) is stale (single 2021 tag), AMD-only, pre-RDNA3/CDNA, image-baked + rebuilt each kernel bump. Without a working reset path a guest reboot/greenboot-failure wedges a passed-through card → host reboot → router drops the network. Newer silicon should migrate to AMD GIM's FLR path. **Now wired into DR (step 12).**
- **R6 — Fail-open network policy.** Headscale's default empty policy is allow-all; the split is only secure once a restrictive Grants policy loads, and huJSON must validate before apply. The SSOT projection + drift-gate **fail-closed** and validate before reload.
- **R7 — Wi-Fi 7 AP maturity.** 802.11be AP mode in hostapd 2.11 is preliminary/driver-gated (ath12k/mt76 + recent kernel), usually a source build baked into the image; many consumer chips STA-only. May fall back to Wi-Fi 6 until drivers mature.
- **R8 — Single-box HA gaps.** headscale HA subnet-routers, FRR VRRP, ceph/gluster distribution all need ≥2 nodes; a lone MiOS-Mini is a single point of failure for guest egress and storage. HA is a fleet property — call it out in any single-node deployment.
- **R9 — Base-image experimental status.** Pinning to Hummingbird today is premature (experimental, distroless-vs-virt collision, young ARK/CKI kernel for GPU vfio). Ship on `fedora-bootc-minimal`, watch Hummingbird; no hard dependency on the zero-CVE base yet.
- **R10 — APU-only / single-iGPU MiOS-Xbox is the worst-served case (NEW, honest).** The split-plane's founding premise assumes ≥1 discrete GPU. On APU-only boxes passthrough is often impossible (R2) and passing the iGPU leaves the host with no console. The named derivative degrades to a documented fallback — single-plane-on-metal or headless-compute-split (§2d) — rather than realizing the motivating win. This is a hardware boundary of the architecture, not a bug to be fixed in software.
- **R11 — GPU fractioning is unavailable on the driver-free host (NEW).** Corrected from the draft's false SR-IOV rescue: mdev *and* SR-IOV both require a host GPU/PF driver (§3a #7), so multi-guest fractioning is impossible while the driver-free-host invariant holds. Operators needing fractioning must consciously break the invariant (install GIM/vGPU-host/xe-SR-IOV on MiOS-Mini) — a loudly-flagged opt-in, not a default capability.
