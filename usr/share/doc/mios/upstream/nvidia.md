# NVIDIA on MiOS

> **Purpose.** MiOS is a single immutable bootc/OCI Fedora workstation image that
> is *also* a local agentic AI OS. Both halves lean on the same discrete NVIDIA
> GPU: a Windows/Linux guest can take it by VFIO passthrough, and the local
> inference lanes can offload to it through CDI. This doc explains how the NVIDIA
> driver gets into the immutable image, how it stays loadable under kernel
> lockdown, and how containers reach the GPU — the foundation everything
> GPU-accelerated on MiOS stands on.
>
> MiOS builds `FROM ghcr.io/ublue-os/ucore-hci:stable-nvidia` (proprietary driver,
> akmod-built, MOK-signed at base-image build time). The NVIDIA Container Toolkit
> plus CDI is the path for containerized GPU workloads.
> Source: `usr/share/doc/mios/concepts/architecture.md` §Pillars / §Hardware-acceleration,
> `usr/share/doc/mios/guides/security.md` §Kargs, `usr/share/doc/mios/upstream/ucore-hci.md`,
> `automation/34-gpu-detect.sh`.

## Where the GPU fits in the whole system

The discrete NVIDIA GPU is a shared resource between MiOS's two faces:

- **Immutable workstation** — GNOME/Wayland uses `nvidia-drm.modeset=1`; KVM/libvirt
  can hand a whole GPU to a guest VM via VFIO passthrough (see §VFIO passthrough),
  and Looking Glass B7 + KVMFR puts that guest's framebuffer back on the host at
  near-native latency.
- **Agentic AI OS** — the local inference lanes offload generation to the same GPU
  through CDI: `mios-llm-light` (`:11450`, the primary llama.cpp lane behind the
  upstream llama-swap proxy image — everyday models, the `mios-opencode` coder model, and
  embeddings), and the gated heavy lanes `mios-llm-heavy` (SGLang, `:11441`) and
  `mios-llm-heavy-alt` (vLLM, `:11440`). These feed the agent stack — agent-pipe
  (`:8640`) → MiOS-Hermes (`:8642`) → pgvector memory — behind one
  OpenAI-compatible endpoint (`MIOS_AI_ENDPOINT`, Architectural Law 5).

The same CDI plumbing serves both paths, and `automation/34-gpu-detect.sh` arbitrates
at boot which GPU is eligible for passthrough versus left for the host/inference.

## Stack

| Layer | Component | Where |
| --- | --- | --- |
| Driver kernel modules | akmod-built, MOK-signed at base-image build time | ucore-hci `:stable-nvidia` |
| User-space libs | `libnvidia-*`, `libcuda` | Installed by ucore-hci |
| Container runtime hook | NVIDIA Container Toolkit | Installed by ucore-hci |
| Device interface | CDI (`/var/run/cdi/`) | Generated by `nvidia-ctk cdi generate`; admin overrides `/etc/cdi/` |
| Wayland modesetting | `nvidia-drm.modeset=1` karg | `usr/lib/bootc/kargs.d/10-nvidia.toml` |
| Unsupported-GPU enable | `nvidia.NVreg_OpenRmEnableUnsupportedGpus=1` karg | `usr/lib/bootc/kargs.d/10-nvidia.toml` |
| Lockdown (module load) | `lockdown=integrity` karg | `usr/lib/bootc/kargs.d/30-security.toml` |
| Memory-init disabled | `init_on_alloc=1`/`init_on_free=1`/`page_alloc.shuffle=1` NOT set globally | CUDA incompatibility — commented in `usr/lib/bootc/kargs.d/01-mios-hardening.toml`; see `guides/security.md` §Kargs |

## akmod & MOK signing

`akmod-nvidia` (in the ucore-hci `:stable-nvidia` base) builds the proprietary
kernel module against the running kernel and signs it with the MOK (Machine Owner
Key) enrolled via Universal Blue. Because MiOS ships kernel lockdown
(`lockdown=integrity` in `30-security.toml`), only signed modules load — and
`module.sig_enforce` is **not** disabled because the MOK-enrolled keys are
sufficient. MiOS doesn't re-sign: the base image's Universal Blue MOK is trusted by
the bootloader chain. This is why NVIDIA on MiOS works under a hardened, locked-down
kernel where unsigned out-of-tree modules would be rejected.

## NVIDIA-safe hardening

MiOS layers SecureBlue-adapted mitigations but filters the NVIDIA-hostile ones:
`init_on_alloc=1`, `init_on_free=1`, and `page_alloc.shuffle=1` are left commented in
`01-mios-hardening.toml` because they interfere with large CUDA allocations and cause
driver instability (uncomment only for CPU-only builds). `lockdown=integrity` (from
the higher-priority `30-security.toml`) deliberately overrides
`01-mios-hardening.toml`'s confidentiality lockdown so MOK-enrolled signed NVIDIA
modules can still load. Full rationale and the per-karg revert table live in
`usr/share/doc/mios/guides/security.md` §Kargs.

## NVIDIA-LTS variant

For older GPUs (Maxwell / Pascal) needing the proprietary 580 LTS driver, override
`MIOS_BASE_IMAGE`:

```bash
MIOS_BASE_IMAGE=ghcr.io/ublue-os/ucore-hci:stable-nvidia-lts just build
```

Note: `:stable-nvidia-lts` is an **ucore-hci tag**, not a separate MiOS image tag.
The non-NVIDIA path (AMD-only / Intel-only hosts) is the upstream `:stable` tag,
likewise selected via `MIOS_BASE_IMAGE`.

## Container GPU access

```bash
# Generate CDI spec for the host's NVIDIA devices
sudo nvidia-ctk cdi generate --output=/var/run/cdi/nvidia.yaml

# Run a CUDA workload via Podman (CDI device ID format)
podman run --rm --device nvidia.com/gpu=all \
  docker.io/nvidia/cuda:12.4.0-base-ubi9 nvidia-smi
```

The CDI spec lists every GPU by UUID; admins can pin specific GPUs to specific
containers via `/etc/cdi/<override>.yaml` (e.g. "GPU 0 for the inference lanes, GPU 1
reserved for VFIO passthrough"). This is exactly how the gated heavy lanes
(`mios-llm-heavy` / `mios-llm-heavy-alt`) are bound to the dGPU once enabled.

- NVIDIA Container Toolkit: <https://github.com/NVIDIA/nvidia-container-toolkit>
- CDI spec: <https://github.com/cncf-tags/container-device-interface>

## VFIO passthrough (MiOS specifically)

For passthrough to a guest VM, see `usr/share/doc/mios/guides/security.md` §Kargs
(IOMMU/VFIO kargs: `iommu=pt`, `amd_iommu=on`, `intel_iommu=on`, the
`13-rtx50-vfio-workaround.toml` reset fix). The host detects passthrough-eligible
devices at runtime via `automation/34-gpu-detect.sh` writing
`/run/mios/gpu-passthrough.status`; the actual logic lives in
`/usr/libexec/mios/gpu-detect`. The boot kargs in `00-mios.toml` blacklist `nouveau`
(`rd.driver.blacklist=nouveau` / `modprobe.blacklist=nouveau`) so the proprietary
driver or VFIO can claim the device cleanly.
