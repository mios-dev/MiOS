Blades & Nodes · cluster topology Distributed · compute fabric

MiOS is a multi-blade, multi-node cluster. A blade is a physical machine; a node is one compute unit on a blade — a CPU, iGPU, dGPU, NPU or other accelerator — together with the inference engines it serves. A node lives local on this blade or remote on another blade, reached over the cluster network. MiOS is forecast to span up to roughly six blades, each carrying several nodes, federated into one agent fabric: the dispatcher fans work across every reachable node by capacity. Each [nodes.<name>] declares WHERE compute lives (endpoint) and WHAT canonical model tag it serves; health_gate lets a come-and-go node auto-join when reachable and drop when gone. The local nodes below ship with MiOS; REMOTE nodes (other blades), an NPU node, phones and edge devices are declared the same way in your /etc/mios/mios.toml overlay (template at the bottom) so no personal blade name / IP is baked into the vendor file.

[nodes.local-dgpu] — discrete GPU node
[nodes.local-cpu] — CPU node
[nodes.local-igpu] — integrated GPU node

Ships endpoint EMPTY: the iGPU is typically served natively on the host (llama.cpp + Vulkan) at a cluster IP set in your overlay. An empty endpoint = inert node.

[nodes.local-vllm] — heavy vLLM lane
[nodes.local-sglang] — heavy SGLang lane
[nodes.local-llamaswap] — llama.cpp multi-model lane
Remote blades · NPU · edge — overlay template (declared in /etc/mios, not vendor)

A remote node on ANOTHER blade — including an NPU node or an edge device — joins the fabric by declaring its own [nodes.<name>] in your overlay. The lane picks the compute bucket (npu for a neural accelerator, accelerator for other ASICs); health_gate = true makes a come-and-go blade auto-join when reachable and drop when gone. Keep real blade names/IPs out of the vendor file — set them here. Example: 

  # second blade's NPU node (overlay)
  [nodes.blade2-npu]
  endpoint    = "http://<blade2-host>:11437/v1"
  model       = "mios-agent"      # the model tag that blade serves
  lane        = "npu"             # neural accelerator compute bucket
  api          = "openai"
  health_gate = true              # auto-join when reachable, drop when gone

  # a remote blade's discrete GPU node (overlay)
  [nodes.blade3-dgpu]
  endpoint    = "http://<blade3-host>:11434/v1"
  model       = "mios-agent"
  lane        = "gpu"
  api          = "ollama"         # or openai / vllm / sglang / llamacpp
  health_gate = true
Users & Tenancy · multi-tenant accounts + ACLs Distributed · multi-tenant

MiOS is multi-tenant. User accounts are PostgreSQL database entries with DB-managed access control, and recall/memory is owner-scoped via row-level security (RLS) so one tenant's data is invisible to another. The human profile the agent uses to address you ([user] name / pronouns / bio) lives in the Personal zone at the top of the page. The knobs here govern who may do what: per-tenant tool ACLs, inbound principal verification, and where RLS is enforced. Per-user tool ceilings are declared as [users.<name>] tables (template below); the RLS mode itself is in the pgvector section ([pgvector].rls_mode = off | enforce).

security — inbound principal / per-tenant auth
[users.<name>] — per-tenant tool ACL (overlay template)

Caps the tool surface by WHO is asking — matched to the principal the chat surface forwards. Same denied_verbs / allowed_verbs / max_permission vocabulary as a per-agent ceiling, plus optional per-user rate/spend quotas. No matching entry ⇒ no restriction (single-tenant MiOS unaffected). Declare in /etc/mios/mios.toml: 

  [users.guest]
  max_permission = "read"          # read-only surface for a guest principal
  denied_verbs   = ["pkg", "shutdown", "reboot"]
  # email        = "guest@example.com"   # also match by forwarded email
  # rpm_limit    = 60               # cap verb dispatches per 60s
  # daily_budget = 5.0              # cap remote-lane spend per 24h

Agents & Federation · sub-agent registry + A2A fabric Distributed · federation

agent-pipe is the always-first orchestrator; Hermes, OpenCode and any peer it can reach are sub-agents it delegates to. Each [agents.<name>] inherits [agents._defaults] then overrides only what differs. The kind field (local-http | remote-http | cli | mobile | edge | node | a2a) is HOW a remote blade or node federates in: remote-http/node reach another blade's OpenAI-compatible endpoint, a2a joins a peer's A2A card. Auth + trust posture make federation zero-trust-capable; A2A peers and epidemic gossip discover the fleet. Array fields (strengths, denied_verbs, allowed_verbs, failover_agents) are comma-separated.

[agents._defaults] — unified template every agent inherits
[agents._defaults.auth] — per-agent credential (zero-trust federation)

An agent presents its OWN credential, so any reachable OpenAI-compatible endpoint joins by network + credential, never bespoke code.

[agents._defaults.trust] — peer trust gate
[agents.hermes] — agent gateway + tool loop
[agents.opencode] — coding sub-agent (CLI shim)
[a2a] — agent-to-agent fleet (cross-blade delegation)

Peer MiOS blades the orchestrator delegates across. List explicit node URLs or set a CIDR to auto-sweep a LAN segment.

[gossip] — epidemic peer discovery (anti-entropy)
[agent_passport] — Open Agent Passport (template; vendor ships commented)

Verifiable, issuer-signed agent identity served at /.well-known/agent-passport.json. EVERY field is optional — the builder fills MiOS defaults. The one activation knob is exposed here; the rest (domain, scope, signing keys, spend ceilings) are set in your overlay (template below). 

  # /etc/mios overlay — activate a signed, verifiable passport
  [agent_passport]
  domain          = "mios.example.net"
  scope           = ["assist.local", "tools.invoke", "web.search"]
  spend_ceiling_amount = 0         # 0 = no autonomous spend authority
  hitl_above_amount    = 0         # human-in-the-loop trigger threshold
  validity_days   = 90
  key_id          = "mios-ap-1"
  signing_key_path = "/etc/mios/agent-passport/ed25519.key"

Deployment targets · output image formats Phase 9 · Deploy
deployment.target_<format>

After the OCI image is built, bootc-image-builder renders one artifact per enabled format under /var/lib/mios/build/output/. Default = qcow2 + iso + anaconda_iso + raw + vhd ON. VMDK / AMI / OVA / GCE / PXE are OFF by default — they need extra host tools (ovftool, aws-cli, etc.). The WSL2 tarball is produced separately (podman save), not through this table.

Windows bootstrap · M:\ partition + dev VM irm|iex Pass-2

Get-MiOS.ps1's Initialize-DataDisk shrinks C:\ and creates the MiOS data partition. Default shrink_mb = 262656 (256 GiB + 512 MB buffer for NTFS reserved space) -- the bare 262144 MB shrink shows up as "255 GB" in Explorer because NTFS reserves ~16 MB at the start.

Passport · Ed25519 agent identity tokens Phase 10 · Runtime

Phase C.3 -- every agent in the stack gets an Ed25519 keypair under key_dir/<agent>/. Security- relevant pgvector writes (tool_call, skill_invocation, event) carry a passport envelope that binds the signature to the exact data. Verification is OFFLINE: public keys are world-readable. Operator can rotate via mios-passport rotate <agent> -- the old kid stays in the agent_keypair table for verifying historical writes.

Routing & domains · 2-stage domain router AIOS · routing

A 2-stage router classifies a query into ONE domain, then shows the planner only that domain's verbs (smaller, faster, more accurate). Daemon & autonomy governor · back-pressure · indexer · post-check AIOS · daemon

mios-daemon runs always-on: it follows logs, indexes the filesystem, and does self-initiated work in the quiet gaps. The back-pressure governor calms autonomous work when the box is hot so it never starves an interactive turn (degrade-open: a probe error treats pressure as absent and proceeds). The post-check map AND-folds tool success with a "did the visible OS effect actually land" signal.
